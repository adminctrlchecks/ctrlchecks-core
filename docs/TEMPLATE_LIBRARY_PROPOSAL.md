## 3. Can the platform actually build these? (node-by-node check against the real registry)

I checked every node type named in the doc against `unified-node-registry.ts` / `node-library.ts` ‚Äî not against the doc's n8n-style naming, against what this platform actually has.

| AI Node (text) | `openai_gpt`, `anthropic_claude`, `google_gemini` | ‚úÖ exists |
| AI Node (Agent, tool access) | `ai_agent` | ‚öÝÔ∏è exists but tool selection is implicit/prompt-driven, not a structured tool-picker UI (the old `tool` connector is marked internal-only/legacy) |
| Code Node | `javascript` / `function` | ‚úÖ exists |
| Send Email | `google_gmail` | ‚úÖ exists |
| Send Message (WhatsApp Business) | `whatsapp` | ‚úÖ exists |
| Send Message (Slack) | `slack_message` | ‚úÖ exists |
| Schedule Trigger | `schedule` | ‚úÖ exists |
| If / branching | `if_else`, `switch` | ‚úÖ exists |
| Chat Trigger | `chat_trigger` | ‚úÖ exists |
| Vector store / knowledge lookup | ‚Äî | ‚ùå **correction (caught during build):** `vector_store`/`embeddings`/`document_ocr`/`invoice_parser`/`resume_parser` exist only as frontend catalog metadata (`ctrl_checks/src/components/workflow/nodeTypes.ts`) ‚Äî none of them are registered in the actual backend execution source of truth (`worker/src/services/nodes/node-library.ts`, which `unified-node-registry.ts` builds `CANONICAL_NODE_TYPES` from via `getAllSchemas()`). | AI vision (classify/read a document) | `google_gemini` (text prompt over the file) | ‚öÝÔ∏è no dedicated vision/OCR node exists in the backend registry either ‚Äî used `google_gemini` node's `prompt` field directly against the uploaded file reference for #3 and #12, instead of a nonexistent `document_ocr` node |

**Bottom line: 12 of 15 workflows are buildable today with zero gaps. 3 workflows (#4, #5, #14, #15 ‚Äî actually 4) have one step each that needs a workaround** (polling instead of an instant DB trigger, or skipping PDF-merge). Nothing blocks shipping all 15 ‚Äî the gaps are "slightly less elegant," not "impossible."

## 6. Other things worth flagging while in here

- **Category dropdown mismatch:** the admin template editor's category dropdown (6 hardcoded options) doesn't match the ~40 category strings actually used in the data. Needs reconciling regardless of this project.
- **`google_forms` node is disconnected from the unified registry** ‚Äî it's not in the canonical schema list, so it runs through a legacy execution path instead of the validated one. Not used by these 15 templates (they use the native `form` node instead), but worth a separate ticket.
- **Supabase node type inconsistency:** registered as `'db'` in one place and `'supabase'` in another. Not blocking for this project since the proposal uses `airtable`/`postgresql`/`google_sheets`, but flagging since "Database" is central to the doc's whole pattern.
- **`ai_agent` has no structured tool-picker UI** ‚Äî workflows #11, #12, #15 lean on this node. It'll function (prompt-driven), but a proper tool-selection UI would make these noticeably more reliable. Worth a fast-follow, not a blocker.
