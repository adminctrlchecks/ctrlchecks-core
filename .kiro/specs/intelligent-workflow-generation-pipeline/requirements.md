# Requirements Document

## Introduction

This document defines the requirements for a complete redesign of the workflow generation pipeline. The current system suffers from four critical issues: repetitive and poorly structured workflow summaries shown to users, node selection that fires indiscriminately rather than only when the AI lacks clarity, backend validation that occurs too late (after the UI has already received a broken graph), and an unclear three-button field ownership control (User / AI-built / AI-runtime) that users do not understand.

The redesigned pipeline is a four-stage system: (1) Prompt Analysis & Intelligent Node Selection, (2) Structural Prompt Generation, (3) Backend Workflow Finalization, and (4) UI Display & User Control. Each stage has a single, well-defined responsibility. The system must work universally for any user prompt and any number of nodes, with no legacy code paths.

---

## Glossary

- **Pipeline**: The end-to-end system that converts a user's natural language prompt into a displayable, executable workflow.
- **Stage**: One discrete phase of the Pipeline with a single responsibility and clear input/output contract.
- **Structural_Prompt**: A human-readable, well-phrased description of the workflow that will be built, including numbered steps, conditions, and data flow — generated after node selection and shown to the user before graph construction begins.
- **Node_Selection_UI**: The interactive screen shown to the user only when the AI cannot confidently determine which integration node to use for a given use case.
- **Workflow_Graph**: The final directed acyclic graph (DAG) of nodes and edges produced by the backend, conforming to the Deterministic Workflow DAG Compiler rules.
- **UnifiedNodeRegistry**: The single source of truth for all node definitions, capabilities, aliases, and behaviors, as defined in `unified-node-registry.ts`.
- **UnifiedGraphOrchestrator**: The single authority for all edge creation, removal, and reconciliation, as defined in `unified-graph-orchestrator.ts`.
- **Field_Ownership**: The classification of each node input field as one of three modes: `User` (user provides the value), `AI-built` (AI fills the value at workflow build time), or `AI-runtime` (AI fills the value at execution time).
- **AI-built_Value**: A field value generated by the AI during workflow construction (build time), stored in the node config.
- **AI-runtime_Value**: A field value that the AI will resolve dynamically at workflow execution time using live context.
- **Capability_Step**: A single ambiguous use case identified during intent analysis for which the AI cannot confidently select a node, requiring user input.
- **DAG**: Directed Acyclic Graph — the structural format all workflows must conform to.
- **Structural_Validation**: The process of verifying that a workflow graph has no broken edges, no orphaned nodes, no duplicate edges, and correct in/out-degree per node type, performed entirely in the backend before the workflow is sent to the UI.

## Requirements

### Requirement 1: Stage-Gated Pipeline Architecture

**User Story:** As a platform engineer, I want the workflow generation pipeline to be divided into four clearly separated stages with explicit contracts between them, so that each stage has a single responsibility and failures are isolated and diagnosable.

#### Acceptance Criteria

1. THE Pipeline SHALL execute exactly four stages in order: Prompt Analysis & Node Selection, Structural Prompt Generation, Backend Workflow Finalization, and UI Display & User Control.
2. WHEN a stage completes, THE Pipeline SHALL pass its output as the sole input to the next stage without any stage reading state from a non-adjacent stage.
3. IF a stage fails, THEN THE Pipeline SHALL halt and return a structured error response identifying the failing stage, its error code, and a human-readable message — without executing subsequent stages.
4. THE Pipeline SHALL expose a `stageTrace` array in every response that records the stage name, duration in milliseconds, input summary, and output summary for each completed stage.
5. THE Pipeline SHALL support any user prompt without requiring prompt-specific code paths, special cases, or hardcoded node type strings outside of `UnifiedNodeRegistry`.

---

### Requirement 2: Intelligent Node Selection (Stage 1 — Prompt Analysis)

**User Story:** As a user, I want the system to automatically select the correct integration nodes for my workflow without interrupting me unless it genuinely cannot determine which node to use, so that simple prompts complete without friction while ambiguous ones get my input.

#### Acceptance Criteria

1. WHEN a user submits a prompt, THE Pipeline SHALL analyze the prompt using `UnifiedNodeRegistry` alias and keyword data to identify candidate node types for each use case in the prompt.
2. WHEN the AI can confidently resolve all use cases to exactly one node type each, THE Pipeline SHALL proceed to Stage 2 without displaying the Node_Selection_UI.
3. WHEN the AI cannot confidently resolve one or more use cases to a single node type, THE Pipeline SHALL produce a `capabilityOptions` array containing one `Capability_Step` per ambiguous use case.
4. WHEN `capabilityOptions` is non-empty, THE Pipeline SHALL pause and present the Node_Selection_UI to the user before proceeding to Stage 2.
5. THE Node_Selection_UI SHALL display each `Capability_Step` with: the use case description, the list of candidate node types with their labels and descriptions, and a clear explanation of which node is appropriate for which scenario.
6. WHEN the user selects nodes in the Node_Selection_UI, THE Pipeline SHALL record the selections as `capabilitySelectionsByStep` and use them as mandatory constraints in all subsequent stages.
7. THE Node_Selection_UI SHALL NOT be shown for prompts where all use cases resolve unambiguously, regardless of the number of nodes in the workflow.
8. THE Pipeline SHALL resolve node type aliases through `UnifiedNodeRegistry.resolveAlias()` — no hardcoded alias maps outside the registry.
9. WHEN a prompt contains multiple distinct use cases, THE Pipeline SHALL identify each use case independently and evaluate node confidence per use case, not per prompt.

---

### Requirement 3: Structural Prompt Generation (Stage 2)

**User Story:** As a user, I want to see a clear, human-readable description of the workflow that will be built — including numbered steps, conditions, and data flow — before the backend constructs the graph, so that I can verify the system understood my intent correctly.

#### Acceptance Criteria

1. WHEN Stage 1 completes (with or without Node_Selection_UI interaction), THE Structural_Prompt_Generator SHALL combine the resolved node types, user intent, and identified use cases into a single Structural_Prompt.
2. THE Structural_Prompt SHALL be written in plain English, with numbered steps, explicit conditions (e.g. "If X, then Y"), and named integrations (e.g. "Gmail", "Slack").
3. THE Structural_Prompt SHALL NOT contain repetitive phrases — each integration or action SHALL appear at most once per logical step.
4. THE Structural_Prompt SHALL NOT use generic headings such as "Review your workflow", "Workflow summary", or "Automation overview".
5. THE Structural_Prompt SHALL clearly communicate: the trigger event, each action step in order, any branching conditions with their outcomes, and the terminal action.
6. THE Structural_Prompt SHALL be shown to the user as a preview before graph construction begins, communicating "this is what we understood and will build".
7. WHEN the user confirms the Structural_Prompt, THE Pipeline SHALL proceed to Stage 3 using the Structural_Prompt as the authoritative specification for graph construction.
8. THE Structural_Prompt SHALL be stored in `workflow.metadata.structuralBlueprintSummary` and passed unchanged to Stage 3 — no modifications after user confirmation.
9. THE Structural_Prompt SHALL have a maximum length of 4000 characters; IF the generated prompt exceeds this limit, THEN THE Structural_Prompt_Generator SHALL truncate it at a sentence boundary and append an ellipsis.

---

### Requirement 4: Backend Workflow Finalization (Stage 3)

**User Story:** As a platform engineer, I want all workflow graph construction, edge wiring, deduplication, and structural validation to happen entirely in the backend before the workflow is sent to the UI, so that the UI always receives a structurally perfect graph and never needs to perform corrections.

#### Acceptance Criteria

1. WHEN Stage 2 completes, THE Backend_Finalizer SHALL construct the complete Workflow_Graph using `UnifiedGraphOrchestrator.initializeWorkflow()` as the sole entry point for graph creation.
2. THE Backend_Finalizer SHALL wire all edges through `UnifiedGraphOrchestrator` — direct mutation of `workflow.edges` (e.g. `workflow.edges.push(...)` or `workflow.edges = [...]`) is forbidden.
3. THE Backend_Finalizer SHALL call `UnifiedGraphOrchestrator.reconcileWorkflow()` after all nodes are placed to correct any stale or broken edges.
4. THE Backend_Finalizer SHALL call `UnifiedGraphOrchestrator.validateWorkflow()` as the final step before sending the workflow to the UI.
5. IF `validateWorkflow()` returns `valid: false`, THEN THE Backend_Finalizer SHALL attempt auto-repair via `UnifiedGraphOrchestrator.reconcileWorkflow()` and re-validate exactly once before returning an error.
6. THE Backend_Finalizer SHALL deduplicate nodes by capability using `UnifiedNodeRegistry` — no two nodes of the same canonical type SHALL appear in the same linear workflow path unless the user's prompt explicitly requires parallel instances.
7. THE Backend_Finalizer SHALL remove orphaned nodes (nodes with no incoming edges except trigger nodes) before sending the workflow to the UI.
8. THE Backend_Finalizer SHALL generate AI-built values for all fields whose `fillMode.default` is `buildtime_ai_once` using the Structural_Prompt as context.
9. THE Backend_Finalizer SHALL configure `Field_Ownership` for every input field of every node, classifying each field as `User`, `AI-built`, or `AI-runtime` according to `UnifiedNodeRegistry` field definitions.
10. WHEN the Workflow_Graph passes validation, THE Backend_Finalizer SHALL attach `workflow.metadata.buildManifest` containing the correlation ID, structural blueprint, authorized node list, and field ownership snapshot.
11. THE Backend_Finalizer SHALL NOT modify the Structural_Prompt or the node chain after Stage 2 has completed — structural format is frozen at Stage 2 output.
12. THE Backend_Finalizer SHALL enforce DAG rules: no cycles, no duplicate node IDs, no duplicate edges, trigger in-degree of 0, trigger out-degree of exactly 1, and terminal node out-degree of 0.

### Requirement 5: UI Display & Three-Button Field Ownership Control (Stage 4)

**User Story:** As a user, I want to see my completed workflow with AI-filled values and a clear three-button control (User / AI-built / AI-runtime) for each field, so that I am the sole decision-maker for how each field value is owned — the system must never automatically change, normalize, or override my ownership selection.

#### Acceptance Criteria

1. WHEN the UI receives the completed Workflow_Graph, THE UI SHALL display the workflow with all nodes properly connected and all AI-built values pre-populated in their respective fields.
2. THE UI SHALL render a three-button control (User / AI-built / AI-runtime) for every input field that supports more than one ownership mode, as determined by the field's `fillMode` definition in `UnifiedNodeRegistry`.
3. WHEN a field's current ownership is `AI-built`, THE UI SHALL display the AI-generated value in the field and highlight the "AI-built" button as active.
4. WHEN a user clicks the "User" button for a field, THE UI SHALL clear the AI-generated value and prompt the user to enter their own value; IF the field is a credential field, THE UI SHALL open the credential connection flow.
5. WHEN a user clicks the "AI-runtime" button for a field, THE UI SHALL mark the field as runtime-resolved and display a placeholder indicating the value will be determined at execution time.
6. WHEN a user clicks the "AI-built" button for a field that was previously set to "User" or "AI-runtime", THE UI SHALL restore the AI-generated value from the build manifest.
7. THE UI SHALL NOT show the three-button control for fields whose `fillMode` supports only one mode (e.g. pure credential fields or pure static config fields).
8. THE UI SHALL arrange all input fields in the order defined by `UnifiedNodeRegistry.inputSchema` for each node type — no arbitrary field ordering.
9. WHEN the user saves the workflow, THE UI SHALL include the current `Field_Ownership` state for every field in the save payload so the backend can persist ownership decisions.
10. THE UI SHALL display the Workflow_Graph as a pure DAG with no broken connections — IF the backend sends a graph with validation errors, THE UI SHALL show an error state rather than rendering a broken graph.
11. AFTER the backend delivers the completed Workflow_Graph to the UI, THE system SHALL NOT automatically change, normalize, or override any field's ownership mode — all ownership changes are exclusively user-initiated via the three-button control.
12. THE backend SHALL NOT re-classify or re-assign `Field_Ownership` values after the workflow has been delivered to the UI — the ownership snapshot in `workflow.metadata.buildManifest` is read-only from that point forward.
13. WHEN a user has not interacted with a field's ownership control, THE UI SHALL preserve the ownership mode set by the backend at build time without modification.

### Requirement 6: Universal Applicability & No Legacy Code Paths

**User Story:** As a platform engineer, I want the redesigned pipeline to work for any user prompt and any combination of nodes without special-casing, so that the system scales to infinite workflows without accumulating technical debt.

#### Acceptance Criteria

1. THE Pipeline SHALL handle any user prompt that produces between 1 and 50 nodes without degradation in correctness or performance.
2. THE Pipeline SHALL resolve all node types exclusively through `UnifiedNodeRegistry` — no hardcoded `if (node.type === '...')` or `switch (node.type)` blocks are permitted outside of `UnifiedNodeRegistry` and `UnifiedGraphOrchestrator`.
3. THE Pipeline SHALL use `UnifiedGraphOrchestrator` as the single authority for all edge operations — no feature code or policy file SHALL directly mutate `workflow.edges`.
4. WHEN the Gemini LLM call fails, THE Pipeline SHALL fall back to a rule-based minimal workflow (manual_trigger → ai_chat_model → log_output) using `UnifiedNodeRegistry` canonical types, without exposing the failure to the user as a hard error.
5. THE Pipeline SHALL NOT contain dual code paths, feature flags, or A/B variants for the core four-stage flow — there is exactly one pipeline.
6. THE Pipeline SHALL apply all node behavior changes made to `UnifiedNodeRegistry` automatically to all future and existing workflows without requiring per-workflow patches.
7. WHEN a new node type is added to `UnifiedNodeRegistry`, THE Pipeline SHALL support it in workflow generation without any changes to pipeline stage code.

---

### Requirement 7: Structural Validation as a Pipeline Contract

**User Story:** As a platform engineer, I want structural validation to be treated as a hard contract that blocks pipeline progression rather than a soft warning, so that broken graphs never reach the UI.

#### Acceptance Criteria

1. THE Pipeline SHALL treat any `validateWorkflow()` result with `valid: false` as a pipeline contract error that blocks the response from being sent to the UI.
2. THE Pipeline SHALL treat the following as hard errors: multiple trigger nodes, orphaned non-trigger nodes, cycles in the graph, edges referencing non-existent node IDs, and switch node out-degree not matching `cases.length`.
3. THE Pipeline SHALL treat the following as warnings (non-blocking): nodes not in execution order, non-required orphaned nodes that were auto-removed, and switch branch count mismatches when the AI generated fewer branch nodes than cases.
4. WHEN a hard validation error occurs after auto-repair, THE Pipeline SHALL return a structured error response with `error: 'ORCHESTRATOR_VALIDATION_FAILED'`, the list of violations, and the stage trace — it SHALL NOT return a broken workflow.
5. THE Pipeline SHALL run `validateWorkflow()` after every structural operation: after `initializeWorkflow()`, after `injectNode()`, after `removeNode()`, and after `reconcileWorkflow()`.
6. THE Pipeline SHALL log all validation errors and warnings to the stage trace with the node IDs and edge IDs involved.

---

### Requirement 8: Workflow Summary Quality

**User Story:** As a user, I want the workflow summary shown in the "Review your workflow" screen to be a clear, non-repetitive, human-readable description of what the workflow does, so that I can quickly understand the automation without reading the same phrase multiple times.

#### Acceptance Criteria

1. THE Structural_Prompt_Generator SHALL produce a summary where each integration (e.g. Gmail, Slack, Google Sheets) appears at most once per logical step.
2. THE Structural_Prompt_Generator SHALL NOT produce summaries containing repeated sentence patterns such as "The data is then passed to X, which does Y. The data is then passed to X, which does Y."
3. THE Structural_Prompt_Generator SHALL use the integration's display name (from `UnifiedNodeRegistry.label`) rather than its internal type string (e.g. "Gmail" not "google_gmail").
4. THE Structural_Prompt_Generator SHALL include the trigger type, all action steps in execution order, any branching logic with labeled outcomes, and the terminal action.
5. WHEN the workflow contains a branching node (if_else or switch), THE Structural_Prompt_Generator SHALL describe each branch outcome on a separate numbered line.
6. THE Structural_Prompt_Generator SHALL produce summaries between 50 and 400 words for workflows with 2–10 nodes.
