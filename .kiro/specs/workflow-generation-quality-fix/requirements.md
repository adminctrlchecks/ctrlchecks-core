# Requirements Document

## Introduction

This feature addresses three interconnected quality problems in the AI workflow generation system:

1. **Node Bloat** — The AI pipeline adds unnecessary nodes (e.g., `text_formatter`, `log_output`) that the user's prompt did not request, violating the minimal-sufficient principle.
2. **AI Build Value Generation** — Config values generated at build time (email subject, body, message content, etc.) are generic placeholders or incorrectly populated instead of being derived from the user's intent and the node's input/output schemas from the `UnifiedNodeRegistry`.
3. **Credential Persistence Bug** — After the AI builds a workflow and the user fills in credentials in the credential panel, clicking "Continue Workflow" loses those credential values because they are not persisted and re-mapped before the confirmation step.

The system uses a `UnifiedNodeRegistry` as the single source of truth for all node definitions, input/output schemas, and credential requirements. Workflow generation flows through a multi-stage AI pipeline: Intent → Node Selection → DSL Generation → DSL Compilation → Graph Orchestration. Credentials flow through `credential-detector.ts` → credential panel UI → `credential-injector.ts` → `workflow-confirm.ts`.

---

## Glossary

- **Node_Selection_Stage**: The pipeline stage in `node-selection-stage.ts` that uses an LLM to select node types from the catalog based on structured intent.
- **UnifiedNodeRegistry**: The single source of truth for all node definitions, input schemas, output schemas, and credential requirements (`unified-node-registry.ts`).
- **Production_Workflow_Builder**: The orchestrator in `production-workflow-builder.ts` that drives DSL generation and compilation.
- **DSL_Compiler**: The `workflow-dsl-compiler.ts` component that compiles a `WorkflowDSL` into an executable workflow graph.
- **Credential_Detector**: The rule-based component in `credential-detector.ts` that scans workflow nodes and identifies required credentials.
- **Credential_Injector**: The component in `credential-injector.ts` that injects user-supplied credential values into workflow node configs.
- **Workflow_Confirm**: The API handler in `workflow-confirm.ts` that processes the "Continue Workflow" action.
- **Credential_Panel**: The UI component that presents required credential fields to the user after workflow generation.
- **Build_Value**: A config field value generated by the AI at workflow build time (e.g., email subject, message body) as opposed to a value resolved at runtime from upstream node output.
- **SelectedNode**: A node type chosen by the `Node_Selection_Stage` LLM, validated against the `UnifiedNodeRegistry`.
- **Minimal_Sufficient_Set**: The smallest set of nodes that fully satisfies the user's stated intent with no extraneous nodes.
- **Pending_Credential_Store**: A transient server-side or client-side store that holds user-entered credential values between the credential panel interaction and the "Continue Workflow" confirmation.

## Requirements

### Requirement 1: Minimal Node Selection

**User Story:** As a workflow author, I want the AI to select only the nodes my prompt explicitly requires, so that the generated workflow is clean, readable, and free of unnecessary steps.

#### Acceptance Criteria

1. WHEN the `Node_Selection_Stage` receives a `StructuredIntent`, THE `Node_Selection_Stage` SHALL select only node types that are directly necessary to fulfill the actions described in the intent.
2. WHEN the `Node_Selection_Stage` produces a `selectedNodes` list, THE `Node_Selection_Stage` SHALL NOT include utility or logging nodes (e.g., `log_output`, `text_formatter`, `set_variable`) unless the user's original prompt explicitly requests them.
3. WHEN the `Node_Selection_Stage` system prompt is constructed, THE `Node_Selection_Stage` SHALL include an explicit instruction stating that the node list must be the minimal sufficient set for the described intent.
4. WHEN the `DSL_Compiler` filters DSL items before compilation, THE `DSL_Compiler` SHALL remove any node whose type is classified as a utility or logging node by the `UnifiedNodeRegistry` and whose config is empty, unless that node type appears in the `SelectedNode` list from the `Node_Selection_Stage`.
5. WHEN the `Production_Workflow_Builder` validates the compiled workflow, THE `Production_Workflow_Builder` SHALL emit a warning for each node present in the workflow that has no corresponding action in the `StructuredIntent`.
6. IF the `Node_Selection_Stage` LLM response contains a node type not present in the `UnifiedNodeRegistry`, THEN THE `Node_Selection_Stage` SHALL discard that node type and log a warning without retrying for that specific type.
7. THE `UnifiedNodeRegistry` SHALL expose a method that returns whether a given node type is classified as a utility or side-effect-only node, so that downstream pipeline stages can query this classification without hardcoding type names.

### Requirement 2: Intent-Driven Build Value Generation

**User Story:** As a workflow author, I want the AI-generated config values (email subject, message body, prompt text, etc.) to reflect my original intent and the data flowing through the workflow, so that the generated workflow is immediately meaningful and requires minimal manual editing.

#### Acceptance Criteria

1. WHEN the `Production_Workflow_Builder` generates config values for a node at build time, THE `Production_Workflow_Builder` SHALL derive text-type field values (fields with `role` of `title_like`, `long_body`, `short_summary`, `prompt`, or `content`) from the user's original prompt text.
2. WHEN the `Production_Workflow_Builder` generates config values for a node at build time, THE `Production_Workflow_Builder` SHALL consult the previous node's output schema from the `UnifiedNodeRegistry` to determine what data fields are available as context for value generation.
3. WHEN the `Production_Workflow_Builder` generates config values for a node at build time, THE `Production_Workflow_Builder` SHALL consult the target node's input schema from the `UnifiedNodeRegistry` to determine the expected type, role, and constraints of each field before generating a value.
4. WHEN a build-time config value is generated for a field whose `fillMode.supportsBuildtimeAI` is `false` in the `UnifiedNodeRegistry`, THE `Production_Workflow_Builder` SHALL leave that field empty or set it to its registry-defined default value rather than generating an AI value.
5. WHEN a build-time config value is generated for a field whose `fillMode.default` is `runtime_ai` in the `UnifiedNodeRegistry`, THE `Production_Workflow_Builder` SHALL NOT populate that field at build time, deferring resolution to the runtime execution stage.
6. WHEN the `Production_Workflow_Builder` generates a build-time value for a `title_like` field (e.g., email subject), THE `Production_Workflow_Builder` SHALL produce a value that is a concise, human-readable summary derived from the user's intent, not a generic placeholder string such as "Generated Subject" or "Process the workflow".
7. WHEN the `Production_Workflow_Builder` generates a build-time value for a `long_body` field (e.g., email body, message content), THE `Production_Workflow_Builder` SHALL produce a value that references the expected upstream data fields by name using the `{{$json.<field>}}` template syntax, based on the previous node's output schema from the `UnifiedNodeRegistry`.
8. IF a node's input schema field has `essentialForExecution` set to `true` in the `UnifiedNodeRegistry` and `fillMode.supportsBuildtimeAI` is `true`, THEN THE `Production_Workflow_Builder` SHALL always generate a non-empty build-time value for that field.
9. THE `UnifiedNodeRegistry` SHALL provide a method that, given a node type and the output schema of the preceding node type, returns a structured context object containing: the available upstream field names, their types, and the target node's field roles — so that build value generation can be schema-driven without hardcoding node-type logic.

### Requirement 3: Credential Persistence and Mapping on Continue Workflow

**User Story:** As a workflow author, I want the credentials I fill in on the credential panel to be saved and correctly mapped into the workflow when I click "Continue Workflow", so that I never lose my credential entries and the workflow is immediately executable.

#### Acceptance Criteria

1. WHEN the user enters a value into a credential field in the `Credential_Panel`, THE `Credential_Panel` SHALL persist that value to the `Pending_Credential_Store` immediately, without waiting for the user to click "Continue Workflow".
2. WHILE the `Credential_Panel` is open, THE `Pending_Credential_Store` SHALL retain all entered credential values keyed by `workflowId` and `provider`, so that values are not lost if the user navigates away and returns.
3. WHEN the user clicks "Continue Workflow", THE `Workflow_Confirm` handler SHALL read all credential values from the `Pending_Credential_Store` for the given `workflowId` before processing the confirmation.
4. WHEN the `Workflow_Confirm` handler reads credentials from the `Pending_Credential_Store`, THE `Credential_Injector` SHALL map each credential value to the correct node config field as defined by the `UnifiedNodeRegistry`'s `credentialSchema` for that node type.
5. WHEN the `Credential_Injector` maps a credential value to a node, THE `Credential_Injector` SHALL use the `credentialSchema` from the `UnifiedNodeRegistry` to determine the correct config field name for each credential field, rather than using a hardcoded field name map.
6. WHEN the `Workflow_Confirm` handler saves the confirmed workflow to the database, THE `Workflow_Confirm` handler SHALL include the credential-injected node configs in the persisted workflow data.
7. IF a required credential field is present in the `Pending_Credential_Store` but its value is an empty string, THEN THE `Workflow_Confirm` handler SHALL return a `400` error response listing the missing credential fields and SHALL NOT persist the workflow as confirmed.
8. WHEN the `Workflow_Confirm` handler successfully persists the workflow with injected credentials, THE `Pending_Credential_Store` SHALL clear the credential entries for that `workflowId`.
9. THE `Credential_Detector` SHALL derive required credential fields for each node type exclusively from the `UnifiedNodeRegistry`'s `credentialSchema`, so that adding a new node type to the registry automatically makes its credentials detectable without modifying `credential-detector.ts`.
10. WHEN the `Credential_Panel` is rendered for a workflow, THE `Credential_Panel` SHALL pre-populate any fields that already have values in the `Pending_Credential_Store` for that `workflowId`, so that previously entered values are visible on re-open.

### Requirement 4: Registry-Driven Utility Node Classification

**User Story:** As a platform engineer, I want utility and logging nodes to be classified in the `UnifiedNodeRegistry` rather than by hardcoded type-name checks, so that the minimal-node and build-value rules apply automatically to any new node type added to the registry.

#### Acceptance Criteria

1. THE `UnifiedNodeRegistry` SHALL support a `isUtilityNode(nodeType: string): boolean` method that returns `true` for nodes whose category or tags indicate they are utility, logging, or side-effect-only nodes.
2. WHEN a new node definition is registered in the `UnifiedNodeRegistry` with a category of `utility` or a tag of `logging`, THE `isUtilityNode` method SHALL return `true` for that node type without requiring any code change outside the registry.
3. THE `Node_Selection_Stage` SHALL call `UnifiedNodeRegistry.isUtilityNode()` to filter candidate nodes rather than maintaining its own list of excluded node type strings.
4. THE `DSL_Compiler` SHALL call `UnifiedNodeRegistry.isUtilityNode()` when deciding whether to filter out empty-config nodes during compilation, rather than checking node type names directly.
