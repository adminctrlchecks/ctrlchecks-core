# Requirements Document

## Introduction

The AI Workflow Generation Engine converts a vague natural language prompt into a fully validated, executable workflow graph. The system operates across three planes of truth: the user content plane (raw prompt), the intent plane (structured representation of what should happen), and the workflow plane (the only executable artifact — a DAG of nodes and edges validated against the registry and orchestrator invariants). Every behavior that depends on what a node is lives in the unified node registry; every behavior that depends on how nodes connect lives in the unified graph orchestrator. AI steers generation — it does not replace registry truth or edge reconciliation.

---

## Glossary

- **System**: The AI Workflow Generation Engine as a whole.
- **Intent_Structurer**: The module that converts a raw user prompt into a `StructuredIntent` object.
- **Summarize_Layer**: The `AIIntentClarifier` service (`summarize-layer.ts`) that produces `WorkflowIntentPlan` and `structuredSummary`.
- **Pipeline_Orchestrator**: The `workflow-pipeline-orchestrator.ts` service that coordinates all pipeline stages and maintains `PipelineContext`.
- **Graph_Orchestrator**: The `unified-graph-orchestrator.ts` service — the single source of truth for all edge creation, removal, and validation.
- **Node_Registry**: The `unified-node-registry.ts` service — the single source of truth for all node behavior, schemas, defaults, and execution.
- **Lifecycle_Manager**: The `workflow-lifecycle-manager.ts` service — the main generation entry point.
- **Fill_Contract**: The per-node field fill semantics generated by `registry-structural-fill-contract.ts` from the Node_Registry.
- **Config_Filler**: The `intelligent-config-filler.ts` service that AI-pre-fills `buildtime_ai_once` fields at workflow creation time.
- **Switch_Planner**: The `switch-case-plan.ts` module that derives switch node cases and expression templates from user intent.
- **StructuredIntent**: A parsed object describing goals, entities, triggers, operations, and constraints derived from the user prompt.
- **WorkflowIntentPlan**: The output of the Summarize_Layer — a structured summary plus an ordered proposed node chain.
- **PipelineContext**: The live state object tracking confidence, expansion, questions, missing fields, and confirmation status throughout the pipeline.
- **FieldFillMode**: One of three values — `manual_static`, `buildtime_ai_once`, or `runtime_ai` — assigned per field in the Node_Registry `inputSchema`.
- **FieldOwnershipClass**: One of three values — `structural`, `value`, or `credential` — assigned per field in the Node_Registry `inputSchema`.
- **AI_Build_Field**: A field whose `fillMode.default` is `buildtime_ai_once`; AI pre-fills it at workflow creation time; user may override.
- **User_Field**: A field whose `fillMode.default` is `manual_static`; user must supply the value.
- **Runtime_Field**: A field whose `fillMode.default` is `runtime_ai`; resolved at execution time from upstream JSON output and intent context.
- **Structured_Prompt**: The clean, concise human-readable sentence or short paragraph produced by the Summarize_Layer that describes the workflow intent and includes the proposed node chain.
- **Node_Chain**: The ordered list of node types (e.g., `manual_trigger → google_sheets → ai_chat_model → google_gmail → log_output`) that the Structured_Prompt commits to building.
- **DAG**: Directed Acyclic Graph — the only valid shape for a workflow graph.
- **Credential_Field**: A field with `ownership = credential`; must be supplied via vault or attach-credentials flow, never AI-hallucinated.
- **Gemini_API**: The Google Gemini API used for all AI operations in this system.
- **validateWorkflow**: The Graph_Orchestrator validation function that must pass after every material graph change.
- **PipelineAnalysis**: A snapshot for UX and debugging — structured prompt, nodes from the structure phase, missing nodes.

## Requirements

### Requirement 1: Structured Prompt Generation from Natural Language

**User Story:** As a user, I want to type a vague natural language prompt and receive a clean, concise structured prompt that describes exactly what workflow will be built, so that I understand the intent before any graph is constructed.

#### Acceptance Criteria

1. WHEN a user submits a natural language prompt, THE Intent_Structurer SHALL parse it into a `StructuredIntent` object containing goals, entities, triggers, operations, and constraints.
2. WHEN the Intent_Structurer produces a `StructuredIntent`, THE Summarize_Layer SHALL generate a `WorkflowIntentPlan` containing a `structuredSummary` and an ordered `proposedNodeChain`.
3. THE Summarize_Layer SHALL produce a `structuredSummary` that is a single sentence or short paragraph — sufficient to describe the workflow intent without omitting key operations and without including registry documentation, boilerplate, or execution-line labels.
4. THE Summarize_Layer SHALL include the `proposedNodeChain` as an ordered list of canonical Node_Registry type strings (e.g., `manual_trigger → google_sheets → ai_chat_model → google_gmail → log_output`).
5. WHEN the Summarize_Layer selects node types, THE Summarize_Layer SHALL resolve all node type strings to canonical registry keys via the unified node type normalizer before including them in the `proposedNodeChain`.
6. IF the Summarize_Layer cannot resolve a node type string to a registered canonical key, THEN THE Summarize_Layer SHALL omit that node type from the `proposedNodeChain` and record it in `PipelineContext.missing_fields`.
7. THE Summarize_Layer SHALL NOT include registry structural fill contract text, planner rule headers, or execution-order labels in the `structuredSummary` returned to the user.

---

### Requirement 2: Conversational Prompt Refinement

**User Story:** As a user, I want to continue the conversation after seeing the structured prompt, so that I can refine the intent and see an updated node chain before the workflow is built.

#### Acceptance Criteria

1. WHEN a user submits a follow-up message after an initial `WorkflowIntentPlan` is shown, THE Pipeline_Orchestrator SHALL merge the follow-up message with the prior `PipelineContext` and re-run intent structuring.
2. WHEN the Pipeline_Orchestrator re-runs intent structuring, THE Intent_Structurer SHALL produce an updated `StructuredIntent` that incorporates all prior answers and the new message.
3. WHEN the same base prompt and the same set of answers are submitted, THE Intent_Structurer SHALL produce a `StructuredIntent` of the same structural class (idempotent refinement).
4. WHEN the Pipeline_Orchestrator updates the `WorkflowIntentPlan`, THE Summarize_Layer SHALL update both the `structuredSummary` and the `proposedNodeChain` to reflect the refined intent.
5. THE Pipeline_Orchestrator SHALL expose the updated `WorkflowIntentPlan` to the client before any graph compilation begins, so the user can review the revised node chain.
6. IF a follow-up message contradicts a prior answer that has already been merged into `PipelineContext`, THEN THE Pipeline_Orchestrator SHALL treat the new message as an override and update `PipelineContext` accordingly.

### Requirement 3: Field Fill Mode Ownership — Three Modes Per Field

**User Story:** As a developer building the system, I want every node field to have a declared fill mode in the Node_Registry, so that the system knows at build time which fields AI pre-fills, which the user must supply, and which are resolved at runtime.

#### Acceptance Criteria

1. THE Node_Registry SHALL assign a `fillMode.default` value of `manual_static`, `buildtime_ai_once`, or `runtime_ai` to every field in every node's `inputSchema`.
2. THE Node_Registry SHALL assign an `ownership` value of `structural`, `value`, or `credential` to every field that requires ownership classification.
3. WHEN a field has `ownership = credential`, THE Node_Registry SHALL set `fillMode.default` to `manual_static` and SHALL NOT allow the Config_Filler to generate a value for that field.
4. THE Fill_Contract SHALL derive its per-node field groupings exclusively from the Node_Registry `inputSchema` — no per-node string overrides are permitted outside the registry.
5. WHEN a new node type is added to the Node_Registry, THE Fill_Contract SHALL automatically include that node's fields in the generated contract section without requiring changes to any other file.
6. FOR ALL node types in the Node_Registry, the Fill_Contract output for a given set of node types SHALL be deterministic — the same input node type list SHALL always produce the same contract text.

### Requirement 4: AI Build — Pre-filling Fields at Workflow Creation Time

**User Story:** As a user, I want AI to pre-fill fields it can determine at workflow creation time (such as email subject lines, switch cases, and conditions), so that I don't have to manually configure every node before running the workflow.

#### Acceptance Criteria

1. WHEN the user confirms the `WorkflowIntentPlan` and triggers workflow generation, THE Config_Filler SHALL pre-fill all fields with `fillMode.default = buildtime_ai_once` for every node in the confirmed `proposedNodeChain`.
2. WHEN the Config_Filler pre-fills a field, THE Config_Filler SHALL use the `structuredSummary`, the `StructuredIntent`, and the upstream node's output schema as context for the Gemini_API call.
3. THE Config_Filler SHALL NOT pre-fill any field with `fillMode.default = manual_static` or `ownership = credential`.
4. WHEN the Config_Filler produces a pre-filled value for a field, THE Config_Filler SHALL validate the value against the field's type and constraints defined in the Node_Registry `inputSchema` before persisting it.
5. IF the Config_Filler receives an invalid or empty response from the Gemini_API for a required `buildtime_ai_once` field, THEN THE Config_Filler SHALL leave the field empty and record it in `PipelineContext.missing_fields` rather than persisting a malformed value.
6. WHEN pre-filled values are stored in the workflow graph, THE System SHALL mark each pre-filled field with its fill mode so the UI can display it as an AI Build field that the user may override.

### Requirement 5: Switch Node Case Generation

**User Story:** As a user, I want the switch node's cases to be automatically derived from my intent (e.g., "route sales, support, general" produces three cases), so that the correct downstream node connects to each case without manual wiring.

#### Acceptance Criteria

1. WHEN a `switch` node appears in the `proposedNodeChain`, THE Switch_Planner SHALL derive the case list and expression template from the user's `StructuredIntent` and original prompt before graph compilation begins.
2. WHEN the Switch_Planner derives cases, THE Switch_Planner SHALL produce at least two distinct case values; IF fewer than two cases can be derived from the prompt, THEN THE Switch_Planner SHALL record the gap in `PipelineContext.missing_fields` and request clarification.
3. THE Switch_Planner SHALL select the discriminant field by querying the upstream node's `outputSchema` in the Node_Registry, preferring fields named `response`, `classification`, `category`, `label`, `result`, `message`, `status`, or `value` in that order.
4. WHEN the Graph_Orchestrator constructs edges for a `switch` node, THE Graph_Orchestrator SHALL create one outgoing edge per case value, labeled `case_1`, `case_2`, … `case_n` in the order the Switch_Planner returned them.
5. WHEN the Graph_Orchestrator wires switch case edges, THE Graph_Orchestrator SHALL connect each `case_n` edge to the downstream node that corresponds to that case as declared in the `WorkflowIntentPlan` — no case edge SHALL be left unconnected.
6. IF the number of switch cases does not match the number of downstream nodes available in the `proposedNodeChain`, THEN THE Graph_Orchestrator SHALL fail validation and surface a pipeline contract error rather than emitting an invalid DAG.

### Requirement 6: Credential Handling — Ask Only When Needed

**User Story:** As a user, I want the system to only ask for credentials when I have explicitly chosen "User" mode for a credential field, so that I am not prompted for secrets I don't need to provide.

#### Acceptance Criteria

1. WHEN the Lifecycle_Manager generates a workflow, THE Lifecycle_Manager SHALL detect required credentials exclusively by reading `credentialSchema.requirements` from the Node_Registry for each node in the confirmed `proposedNodeChain`.
2. WHEN all credential fields for a node are in `buildtime_ai_once` or `runtime_ai` fill mode, THE Lifecycle_Manager SHALL NOT prompt the user for credentials for that node.
3. WHEN a user explicitly sets a credential field to `manual_static` mode, THE Lifecycle_Manager SHALL add that credential to the `requiredCredentials` list and surface the attach-credentials flow for that field only.
4. THE Lifecycle_Manager SHALL NOT include credential field values in any Gemini_API prompt, summarizer prompt, or log entry at default verbosity.
5. WHEN the attach-credentials flow is triggered, THE System SHALL route credential values through the vault or `attach-credentials` API endpoint — credential values SHALL NOT be stored in `node.data.config` as plain text.
6. IF a workflow contains nodes whose credential fields are all resolved via vault or runtime, THEN THE System SHALL open the workflow directly without displaying a credential prompt screen.

---

### Requirement 7: Graph Construction via Unified Graph Orchestrator

**User Story:** As a developer, I want all workflow edges to be created exclusively through the unified graph orchestrator, so that no feature code can produce an invalid DAG by hand-wiring edges.

#### Acceptance Criteria

1. WHEN the Lifecycle_Manager constructs a new workflow from a confirmed `proposedNodeChain`, THE Lifecycle_Manager SHALL call `Graph_Orchestrator.initializeWorkflow(nodes)` as the sole method for creating the initial edge set.
2. WHEN a node is added to an existing workflow during lifecycle resolution, THE Lifecycle_Manager SHALL call `Graph_Orchestrator.injectNode(workflow, node, context)` — direct mutation of `workflow.edges` is forbidden.
3. WHEN a node is removed from a workflow, THE Lifecycle_Manager SHALL call `Graph_Orchestrator.removeNode(workflow, nodeId)` — direct splicing of `workflow.nodes` or `workflow.edges` is forbidden.
4. AFTER every material graph change (initializeWorkflow, injectNode, removeNode, reconcileWorkflow), THE Lifecycle_Manager SHALL call `Graph_Orchestrator.validateWorkflow(workflow)` and SHALL treat any validation failure as a pipeline contract error that blocks workflow save.
5. THE Graph_Orchestrator SHALL derive all edge decisions from the Node_Registry — branching ports, merge semantics, and terminal tags — without any `if (node.type === '...')` string comparisons in the orchestrator or lifecycle code.
6. WHEN `Graph_Orchestrator.validateWorkflow` is called, THE Graph_Orchestrator SHALL verify: exactly one trigger, no orphan nodes, no cycles, correct in-degree and out-degree per node category, and edge labels matching registry-declared outgoing ports.

### Requirement 8: Runtime AI Field Resolution

**User Story:** As a user, I want fields marked as runtime AI to be resolved automatically at execution time using upstream node output and workflow intent, so that I don't need to manually map data between nodes.

#### Acceptance Criteria

1. WHEN a node with `runtime_ai` fields executes, THE Node_Registry `execute()` function SHALL resolve those fields by passing the upstream node's JSON output and the workflow's `structuredSummary` to the Gemini_API.
2. WHEN the Gemini_API resolves a `runtime_ai` field, THE Node_Registry `execute()` function SHALL validate the resolved value against the field's type constraints before using it.
3. THE Node_Registry `execute()` function SHALL resolve all `{{$json.*}}` template expressions via the universal template resolver before passing config to the legacy executor.
4. IF the Gemini_API returns an empty or invalid value for a required `runtime_ai` field at execution time, THEN THE Node_Registry `execute()` function SHALL return a `NodeExecutionResult` with `success: false` and a descriptive error code.
5. THE output of a `runtime_ai` resolution SHALL be pure JSON or plain text — mixed formats that embed registry documentation or template syntax SHALL be rejected.
6. WHEN a `runtime_ai` field is resolved at execution time, THE System SHALL NOT re-use a cached value from a prior execution run unless the upstream output and intent context are identical.

### Requirement 9: Node Chain UX — Field Mode Toggling

**User Story:** As a user, I want to see and toggle the fill mode for each node field (Not Necessary / User / AI Build / Runtime), so that I have full control over which fields AI manages and which I configure myself.

#### Acceptance Criteria

1. THE System SHALL expose a field mode toggle for every field in every node in the confirmed workflow, with states: Not Necessary, User (`manual_static`), AI Build (`buildtime_ai_once`), and Runtime (`runtime_ai`).
2. WHEN a field's toggle state is "Not Necessary", THE System SHALL not display an input for that field and SHALL not include it in the credential prompt or configuration wizard.
3. WHEN a user sets a field's toggle to "User", THE System SHALL display an input for that field and SHALL add it to the configuration wizard for the user to fill.
4. WHEN a user sets a field's toggle to "AI Build", THE System SHALL display the AI-pre-filled value for that field and SHALL allow the user to edit it before saving.
5. WHEN a user sets a field's toggle to "Runtime", THE System SHALL display a read-only indicator that the field will be resolved at execution time and SHALL not require user input.
6. WHEN a field has `ownership = credential` and toggle state is "User", THE System SHALL route the credential value through the vault attach-credentials flow rather than storing it in the node config directly.
7. THE System SHALL persist the user's toggle selection per field in the saved workflow graph so that re-opening the workflow restores the same toggle states.

### Requirement 10: Intent Sanitization — Preventing Contract Leakage

**User Story:** As a developer, I want the system to strip registry fill contract text and boilerplate from any text used for form field extraction, so that downstream consumers never mistake template documentation for user-supplied data keys.

#### Acceptance Criteria

1. WHEN the Pipeline_Orchestrator assembles a combined text blob for form field extraction, THE Pipeline_Orchestrator SHALL pass it through `sanitizeIntentTextForFormFieldExtraction` before any field key discovery runs.
2. THE sanitizer SHALL strip all sections that begin with registry fill contract headers (e.g., "## Configuration contract", "Semantics (universal):", "Planner rules:") from the combined text.
3. THE sanitizer SHALL strip all execution-order slugs, node type labels in parentheses, and boilerplate ownership annotation lines from the combined text.
4. WHEN the sanitizer strips a section, THE sanitizer SHALL NOT alter the remaining user-supplied content — only registry-generated fragments SHALL be removed.
5. FOR ALL inputs to the sanitizer that contain no registry contract text, THE sanitizer SHALL return the input unchanged (idempotent on clean input).
6. THE System SHALL include at least one test fixture that contains a known registry fill contract fragment and asserts that the sanitizer removes it without altering adjacent user content.

### Requirement 11: Pipeline Confidence and Missing Field Tracking

**User Story:** As a developer, I want the pipeline to track confidence scores and missing fields throughout generation, so that low-confidence or incomplete intents are surfaced to the user rather than silently producing broken workflows.

#### Acceptance Criteria

1. THE Pipeline_Orchestrator SHALL compute a confidence score for the `StructuredIntent` after each intent structuring pass and store it in `PipelineContext.confidence`.
2. WHEN `PipelineContext.confidence` falls below the system-defined threshold, THE Pipeline_Orchestrator SHALL set `PipelineContext.requires_confirmation = true` and SHALL NOT proceed to graph compilation without user confirmation.
3. WHEN the pipeline identifies a field required for structural validity that cannot be derived from the prompt, THE Pipeline_Orchestrator SHALL add that field to `PipelineContext.missing_fields`.
4. WHEN `PipelineContext.missing_fields` is non-empty after intent structuring, THE Pipeline_Orchestrator SHALL generate the minimum set of clarification questions needed to resolve those fields before proceeding.
5. WHEN `Graph_Orchestrator.validateWorkflow` fails after graph compilation, THE Pipeline_Orchestrator SHALL record the validation failure as a first-class reason in `PipelineContext` and SHALL surface it to the client — a low confidence score alone SHALL NOT be the only signal.
6. THE Pipeline_Orchestrator SHALL expose `PipelineAnalysis` (structured prompt, nodes from structure phase, missing nodes) as a separate snapshot from `PipelineContext` so UX and debugging consumers can read them independently.

### Requirement 12: Registry-Driven Node Selection — No Hardcoding

**User Story:** As a developer, I want node selection, ordering, and edge wiring to be driven entirely by the Node_Registry and Graph_Orchestrator, so that adding a new node type never requires changes outside the registry.

#### Acceptance Criteria

1. THE Summarize_Layer SHALL select candidate node types by querying the Node_Registry for `aiSelectionCriteria.keywords`, `tags`, and `aliases` — no hardcoded node type strings SHALL appear in the selection logic outside the registry.
2. WHEN the Summarize_Layer or Pipeline_Orchestrator normalizes a node type string, THE System SHALL use the unified node type normalizer and SHALL treat any spelling variant (e.g., `gmail` vs `google_gmail`) as a normalization bug to be fixed in the normalizer, not as two valid parallel paths.
3. THE Graph_Orchestrator SHALL determine branching port names (`true`/`false`, `case_1`…`case_n`) by reading `outgoingPorts` and `isBranching` from the Node_Registry — no port name SHALL be hardcoded in the orchestrator or lifecycle code.
4. WHEN a new node type is registered in the Node_Registry with `workflowBehavior.alwaysRequired = true`, THE Lifecycle_Manager SHALL automatically include that node in every generated workflow without requiring changes to any other file.
5. WHEN a new node type is registered with `workflowBehavior.alwaysTerminal = true`, THE Graph_Orchestrator SHALL enforce that node as a terminal (out-degree 0) in every workflow that contains it.
6. THE System SHALL include a golden-path test for each new integration node: prompt → `StructuredIntent` → `WorkflowIntentPlan` → graph compilation → `validateWorkflow` passes with zero structural errors.

---

### Requirement 13: Workflow Validation as Pipeline Gate

**User Story:** As a developer, I want workflow validation to be a hard gate that blocks saving or executing any workflow with structural errors, so that users never encounter broken graphs at runtime.

#### Acceptance Criteria

1. THE Lifecycle_Manager SHALL call `Graph_Orchestrator.validateWorkflow` after DSL compilation, after minimal workflow policy application, after safety node injection, and after any lifecycle adjustment that adds or removes nodes.
2. WHEN `Graph_Orchestrator.validateWorkflow` returns `valid: false`, THE Lifecycle_Manager SHALL not persist the workflow and SHALL return a structured error to the client that identifies the specific violated invariant.
3. THE Graph_Orchestrator SHALL enforce the following invariants in `validateWorkflow`: exactly one trigger node, all non-trigger nodes reachable from the trigger, no cycles, trigger out-degree exactly 1, normal action node in-degree exactly 1 and out-degree exactly 1, `if_else` out-degree exactly 2 with ports `true` and `false`, `switch` out-degree matching the number of declared cases with ports `case_1`…`case_n`, terminal node out-degree 0.
4. WHEN a validation invariant is violated, THE Graph_Orchestrator SHALL return a descriptive error message that names the offending node ID, the violated rule, and the expected vs actual degree or port label.
5. THE System SHALL not soften DAG compiler rules in any Gemini_API prompt to "make it work" — IF the AI proposal violates a DAG rule, THEN THE Lifecycle_Manager SHALL reject the proposal and re-run generation with a corrected constraint, not relax the invariant.
6. WHEN the Lifecycle_Manager detects that a generated graph fails validation on the first attempt, THE Lifecycle_Manager SHALL attempt one internal repair pass via `Graph_Orchestrator.reconcileWorkflow` before surfacing the error to the client.

### Requirement 14: Structured Prompt Round-Trip Consistency

**User Story:** As a developer, I want the structured prompt and node chain to remain consistent with the compiled graph, so that what the user sees before building matches what actually gets built.

#### Acceptance Criteria

1. WHEN the Lifecycle_Manager compiles a graph from a confirmed `WorkflowIntentPlan`, THE compiled graph SHALL contain exactly the node types listed in `proposedNodeChain` plus any nodes added by `workflowBehavior.alwaysRequired` policy — no additional node types SHALL be silently inserted.
2. WHEN the compiled graph differs from the `proposedNodeChain` due to policy additions, THE Lifecycle_Manager SHALL update the `structuredSummary` to reflect the final node set before returning the result to the client.
3. THE `structuredSummary` returned to the client SHALL NOT be used as input to any graph compilation step — graph compilation SHALL consume `StructuredIntent` and `proposedNodeChain` as structured data, not the prose summary.
4. FOR ALL valid `WorkflowIntentPlan` objects, compiling the plan to a graph and then extracting the node type list from the graph SHALL produce a list equivalent to `proposedNodeChain` plus policy-required nodes (round-trip consistency).
5. WHEN the user edits the `structuredSummary` text before confirming, THE System SHALL re-parse the edited text through the Intent_Structurer to produce an updated `StructuredIntent` before proceeding to graph compilation.

### Requirement 15: Observability and Debug Artifacts

**User Story:** As a developer, I want every generation run to produce structured debug artifacts, so that I can trace exactly what happened at each pipeline stage without relying on free-form log text.

#### Acceptance Criteria

1. THE Pipeline_Orchestrator SHALL produce a `PipelineContext` object for every generation run that includes: confidence score, `missing_fields` list, clarification questions asked, `requires_confirmation` flag, and the final `StructuredIntent`.
2. THE Pipeline_Orchestrator SHALL produce a `PipelineAnalysis` snapshot for every generation run that includes: the `structuredSummary`, the node list from the structure phase, and any nodes that were in `proposedNodeChain` but absent from the compiled graph.
3. WHEN a generation run fails at any pipeline stage, THE Pipeline_Orchestrator SHALL include the last known `StructuredIntent` or cached `WorkflowIntentPlan` in the error response so the client can offer a recovery UI.
4. THE System SHALL log prompt content at debug level only — at info level, THE System SHALL log only prompt length, confidence score, node chain length, and validation result.
5. WHEN the Gemini_API call fails mid-pipeline, THE Pipeline_Orchestrator SHALL preserve the last valid `PipelineContext` state, log the failure with a structured error code, and return an explicit error to the client rather than silently falling back to a different intent path.

### Requirement 16: Rich Structural Prompt with Field-Level Detail

**User Story:** As a user, I want the structural prompt to show me exactly what each node will contain — including form fields, conditions, cases, operations, and data flow — so I have a complete blueprint of the workflow before it is built.

#### Acceptance Criteria

1. WHEN the Summarize_Layer produces a `WorkflowIntentPlan`, the `structuredSummary` SHALL include a per-node description block for every node in `proposedNodeChain`.
2. FOR a `form` node, the per-node description SHALL list every input field that will be created, including field name, type (text/number/email/select/etc.), and whether it is required.
3. FOR an `if_else` node, the per-node description SHALL state the condition expression, which upstream field it evaluates, the true-branch destination node, and the false-branch destination node.
4. FOR a `switch` node, the per-node description SHALL list every case value, the discriminant field being evaluated, and the downstream node each case routes to.
5. FOR integration nodes (gmail, google_sheets, slack, etc.), the per-node description SHALL state the operation (e.g., "send email", "read rows", "post message"), the data source for each input field, and the purpose of this node in the workflow.
6. FOR every node, the per-node description SHALL state what data it receives from upstream and what data it passes downstream.
7. THE `structuredSummary` SHALL be human-readable prose — not JSON, not code, not registry documentation — that a non-technical user can understand.
8. THE Summarize_Layer SHALL derive all field-level details from the Node_Registry `inputSchema` and the user's `StructuredIntent` — no hardcoded field lists are permitted.

### Requirement 17: Same-Node-Type Branch Handling

**User Story:** As a user, I want workflows where both branches of a condition use the same node type (e.g., both true and false paths send a Gmail) to produce two separate, correctly configured node instances — one per branch — so each branch does something different.

#### Acceptance Criteria

1. WHEN the `proposedNodeChain` contains an `if_else` node followed by the same node type on both the true and false paths, the Graph_Orchestrator SHALL create two separate node instances with distinct IDs.
2. WHEN the Config_Filler pre-fills `buildtime_ai_once` fields for same-type branch nodes, it SHALL use the branch intent (true-path purpose vs false-path purpose) as context to produce different configs for each instance.
3. THE `structuredSummary` SHALL explicitly describe what each branch node does differently (e.g., "True branch: Gmail sends approval email with subject 'Application Approved'; False branch: Gmail sends rejection email with subject 'Application Not Approved'").
4. WHEN the Graph_Orchestrator wires edges for an `if_else` node, the `true` edge SHALL connect to the true-branch node instance and the `false` edge SHALL connect to the false-branch node instance — never to the same node.
5. FOR switch nodes, the same rule applies: each case SHALL connect to a separate node instance even when multiple cases use the same node type.
6. THE Node_Registry SHALL not be modified to support this — the Graph_Orchestrator and Config_Filler SHALL handle multi-instance creation generically for any node type.

### Requirement 18: Senior AI / Junior AI Two-Tier Reasoning Architecture

**User Story:** As a developer, I want the AI pipeline to use a two-tier reasoning architecture where a Senior AI validates every stage transition and a Junior AI handles stage execution, so that the system self-corrects before producing incorrect outputs.

#### Acceptance Criteria

1. THE pipeline SHALL implement a Senior AI role that holds the full context: original user prompt, `StructuredIntent`, `WorkflowIntentPlan`, registry knowledge summary, and all prior stage outputs.
2. THE pipeline SHALL implement a Junior AI role that executes individual pipeline stages (node selection, field derivation, condition generation, case extraction) and submits its proposed output to the Senior AI for approval before proceeding.
3. WHEN the Junior AI proposes a node selection, the Senior AI SHALL validate: (a) all selected node types exist in the registry, (b) the selection is the minimum sufficient set for the user's intent, (c) no node type is duplicated unless the intent requires multiple instances, (d) the proposed chain forms a valid DAG.
4. WHEN the Senior AI rejects a Junior AI proposal, the Senior AI SHALL return a structured correction with the specific reason and the corrected value — the Junior AI SHALL incorporate the correction and re-submit.
5. THE Senior AI SHALL recall all prior stage outputs when validating a new stage — it SHALL NOT validate a stage in isolation.
6. WHEN the Junior AI has received Senior AI approval for all stages, the pipeline SHALL proceed to graph compilation — no stage SHALL be compiled without Senior AI approval.
7. THE Senior AI and Junior AI SHALL both use the Gemini API — the Senior AI uses `gemini-2.5-pro` for its validation calls; the Junior AI uses `gemini-2.5-flash` for its execution calls.
8. THE two-tier architecture SHALL be implemented as a reusable `PipelineReasoningCoordinator` class that any pipeline stage can use — no stage-specific Senior/Junior logic is permitted outside this class.

### Requirement 19: Self-Healing Before Credential Step

**User Story:** As a user, I want the workflow graph to be fully validated and self-healed before I am ever asked for credentials, so that I never provide credentials for a broken workflow.

#### Acceptance Criteria

1. THE Lifecycle_Manager SHALL call `Graph_Orchestrator.validateWorkflow` and, if invalid, `Graph_Orchestrator.reconcileWorkflow` BEFORE evaluating any credential requirements.
2. WHEN `reconcileWorkflow` produces a valid graph, THE Lifecycle_Manager SHALL proceed to the credential step with the repaired graph.
3. WHEN `reconcileWorkflow` cannot produce a valid graph after one repair attempt, THE Lifecycle_Manager SHALL return a structured error to the client and SHALL NOT proceed to the credential step.
4. THE self-healing pass SHALL verify: all branch nodes have the correct number of outgoing edges, all terminal nodes have out-degree 0, all non-trigger nodes are reachable from the trigger, no cycles exist.
5. WHEN self-healing repairs a graph, THE Lifecycle_Manager SHALL update the `structuredSummary` to reflect any structural changes made during repair.
6. THE credential step SHALL only be reached when `Graph_Orchestrator.validateWorkflow` returns `valid: true` — this is a hard gate, not a soft warning.

### Requirement 20: Legacy Keyword Hardcoding Removal

**User Story:** As a developer, I want all hardcoded keyword-to-node-type mappings removed from the codebase and replaced with pure Gemini + registry reasoning, so that adding a new node type never requires updating keyword maps.

#### Acceptance Criteria

1. THE `enhanced-keyword-matcher.ts` hardcoded `keywordVariations` map SHALL be removed; all keyword matching SHALL use the Node_Registry `aiSelectionCriteria.keywords`, `tags`, and `aliases` fields exclusively.
2. WHEN the Gemini-first node selection path (`gemini-node-selector.ts`) returns a non-empty result, THE system SHALL use that result exclusively and SHALL NOT fall back to keyword matching.
3. WHEN the Gemini-first path returns an empty result, THE system SHALL use registry-based keyword matching (from `aiSelectionCriteria.keywords` and `tags`) as the fallback — never hardcoded maps.
4. THE `logic-intent safeguard` in `summarize-layer.ts` that hardcodes `['form', 'javascript', 'if_else', 'function']` SHALL be replaced with a registry query that finds all nodes tagged with `logic` or `conditional` category.
5. AFTER removing legacy code, THE system SHALL pass all existing golden-path tests without modification to test fixtures.
6. THE removal SHALL be done AFTER the new system is implemented and verified — not before.

### Requirement 21: Precise Node Quantity Selection

**User Story:** As a user, I want the AI to select exactly the right number of nodes for my intent — no unnecessary extras, no missing ones — so that the generated workflow is clean and directly matches what I asked for.

#### Acceptance Criteria

1. WHEN the Senior AI validates a node selection, it SHALL apply a sufficiency check: for each selected node, there SHALL be a clear reason derived from the user's `StructuredIntent` why that node is needed.
2. WHEN a node has no clear reason derived from the intent, the Senior AI SHALL instruct the Junior AI to remove it from the selection.
3. THE pipeline SHALL NOT add `log_output` nodes unless the user's intent explicitly mentions logging, monitoring, or observability — OR unless `workflowBehavior.alwaysTerminal` is set in the registry.
4. WHEN the user's intent implies a single linear action (e.g., "send an email when a form is submitted"), the pipeline SHALL produce exactly the minimum nodes: one trigger + one action node (+ any `alwaysRequired` nodes from the registry).
5. THE Senior AI sufficiency check SHALL be logged in `PipelineContext` as `nodeSelectionRationale`: a map of nodeType → reason string for every node in the final selection.
6. WHEN the same node type appears multiple times in the selection, each instance SHALL have a distinct reason in `nodeSelectionRationale`.
