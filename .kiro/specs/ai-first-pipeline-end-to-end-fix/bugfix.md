# Bugfix Requirements Document

## Introduction

The AI-first workflow generation pipeline is entirely broken end-to-end. Eight distinct defects span every stage of the pipeline: the node catalog is sent twice causing token overflow and JSON parse failures; node hydration stores defaults in a dead property instead of `node.data.config`; edge reasoning ignores the LLM's proposed ordering so the orchestrator silently recomputes its own; validation sends a summary string instead of the actual graph so the LLM cannot evaluate data flow or semantic alignment; Stage 2 (structural prompt generation) is completely absent so downstream stages have no shared blueprint context; credential discovery is never called so the API response contains no credential data; the field ownership map is never extracted so the UI cannot render the input configuration panel; and `gemini-orchestrator` caps `workflow-generation` responses at 4,000 tokens causing silent truncation on any non-trivial workflow.

The combined effect is that no workflow generated by this pipeline is structurally correct, semantically validated, or usable by the UI without manual intervention.

Three additional UI-layer defects have been identified from observed frontend behavior: the ComprehensiveQuestions step is skipped entirely â€” the "Refining Workflow Plan" overlay appears before the user has answered any questions; the Field Ownership step renders empty ("No fields in this category") despite the backend producing a valid `fieldOwnershipMap` with 24 fields; and credential questions generated by the pipeline leak into the questions UI step instead of being confined to the dedicated Credentials step.

## Bug Analysis

### Current Behavior (Defect)

1.1 WHEN the node-selection stage builds its LLM request THEN the system sends the full node catalog in both the system prompt and the message body, doubling token consumption and causing the LLM to truncate or fail JSON parsing.

1.2 WHEN the pipeline hydrates selected nodes with registry defaults THEN the system stores the defaults in a `_hydratedConfig` property on the `SelectedNode` object and never writes them to `node.data.config`, leaving every node with an empty config when it reaches the orchestrator.

1.3 WHEN the edge-reasoning stage calls `unifiedGraphOrchestrator.initializeWorkflow()` THEN the system passes only the node list and omits the LLM's `orderedNodes` as `initialExecutionOrder`, causing the orchestrator to recompute its own execution order and discard the AI's intended sequencing.

1.4 WHEN the validation stage sends the workflow to the LLM THEN the system constructs the message as `"nodes=N, edges=M, node_types=..."` â€” a summary string â€” instead of the actual graph JSON, making it impossible for the LLM to evaluate data flow coherence or semantic alignment.

1.5 WHEN the pipeline sequences its stages THEN the system has no Stage 2 (structural prompt generation) between intent extraction and node selection, so no shared blueprint context is generated and downstream stages receive no explanation of how the workflow should work.

1.6 WHEN the pipeline produces a validated workflow THEN the system never calls `credentialDiscoveryPhase.discoverCredentials()`, so `AiPipelineOutput` contains no `requiredCredentials` or `missingCredentials` data and the UI credential wizard cannot be shown.

1.7 WHEN the pipeline produces a validated workflow THEN the system never extracts the `fillMode` defaults from each node's `inputSchema` in the registry, so `AiPipelineOutput` contains no field ownership map and the UI cannot render the input field configuration panel.

1.8 WHEN `gemini-orchestrator` handles a `workflow-generation` request THEN the system applies a `maxTokens` cap of 4,000, causing the LLM response to be silently truncated for any workflow with more than a few nodes, producing malformed JSON.

### Expected Behavior (Correct)

2.1 WHEN the node-selection stage builds its LLM request THEN the system SHALL include the node catalog only in the system prompt and SHALL NOT repeat it in the message body, keeping the message body to structured intent only.

2.2 WHEN the pipeline hydrates selected nodes with registry defaults THEN the system SHALL write the `defaultConfig()` output directly to `node.data.config` on each `WorkflowNode` before passing nodes to the edge-reasoning stage.

2.3 WHEN the edge-reasoning stage calls `unifiedGraphOrchestrator.initializeWorkflow()` THEN the system SHALL convert the LLM's `orderedNodes` list into an `ExecutionOrder` object and pass it as `initialExecutionOrder`, so the orchestrator respects the AI's sequencing.

2.4 WHEN the validation stage sends the workflow to the LLM THEN the system SHALL serialize the actual `workflow.nodes` and `workflow.edges` arrays as JSON in the message body so the LLM can evaluate all four validation dimensions.

2.5 WHEN the pipeline sequences its stages THEN the system SHALL execute a structural-prompt stage between intent extraction and node selection that generates a plain-language blueprint of the workflow, which is passed as shared context to all downstream stages.

2.6 WHEN the pipeline produces a validated workflow THEN the system SHALL call `credentialDiscoveryPhase.discoverCredentials(workflow, userId)` and include `requiredCredentials` and `missingCredentials` in `AiPipelineOutput` and in the `generate-workflow.ts` API response.

2.7 WHEN the pipeline produces a validated workflow THEN the system SHALL walk each node's `inputSchema` from the registry, extract the `fillMode.default` per field, and include a `fieldOwnershipMap` (keyed by nodeId, then fieldName) in `AiPipelineOutput` and in the API response.

2.8 WHEN `gemini-orchestrator` handles a `workflow-generation` request THEN the system SHALL apply a `maxTokens` cap of at least 16,000 to accommodate full workflow JSON responses without truncation.

### Unchanged Behavior (Regression Prevention)

3.1 WHEN the intent stage runs THEN the system SHALL CONTINUE TO extract `StructuredIntent` from the user prompt via LLM with no keyword pre-filter, exactly as it does today.

3.2 WHEN the node-selection stage validates LLM output THEN the system SHALL CONTINUE TO discard unknown node types against the registry and return `NO_VALID_NODES` error when zero valid types remain.

3.3 WHEN the edge-reasoning stage detects a cycle in the LLM's proposed edges THEN the system SHALL CONTINUE TO re-prompt the LLM exactly once with the cycle path identified before returning `CYCLE_DETECTED`.

3.4 WHEN the validation stage completes THEN the system SHALL CONTINUE TO call `unifiedGraphOrchestrator.validateWorkflow()` as a structural safety net regardless of LLM validation outcome.

3.5 WHEN the validation stage finds error-severity issues THEN the system SHALL CONTINUE TO attempt exactly one repair pass before returning the partially repaired graph with remaining issues.

3.6 WHEN `generate-workflow.ts` receives a request THEN the system SHALL CONTINUE TO invoke `AiFirstPipeline` directly with no feature flag, no conditional branching, and no fallback to any legacy pipeline.

3.7 WHEN the pipeline returns a workflow THEN the system SHALL CONTINUE TO return the workflow even if credentials are missing, so the UI can show the credential wizard without blocking the workflow response.

3.8 WHEN any stage calls `unifiedGraphOrchestrator` THEN the system SHALL CONTINUE TO never write to `workflow.edges` directly â€” all edge mutations go through the orchestrator only.

## UI Bug Analysis

### Current Behavior (Defect) â€” UI Layer

1.9 WHEN the pipeline backend completes (including `ai_pipeline_validation_parse_failed` fallback via orchestrator safety net) THEN the frontend immediately renders the "Refining Workflow Plan" overlay on top of the ComprehensiveQuestions UI, so the user never sees or answers the 24 generated questions.

1.10 WHEN the Field Ownership step renders THEN the system displays "No fields in this category for this workflow" for all categories despite the backend `field_ownership` stage having produced a `fieldOwnershipMap` with `nodes=4, fields=24` â€” the UI component does not consume the `fieldOwnershipMap` data from the pipeline response.

1.11 WHEN the ComprehensiveQuestions UI step renders its question list THEN the system includes credential questions (e.g., `credentialId` for `google_gmail`, `askOrder: 0.5`) alongside node-field questions, causing credential prompts to appear in the wrong step instead of the dedicated Credentials step.

### Expected Behavior (Correct) â€” UI Layer

2.9 WHEN the pipeline backend completes workflow generation THEN the system SHALL display the ComprehensiveQuestions UI and wait for the user to answer all non-credential questions before transitioning to the "Refining Workflow Plan" loading state.

2.10 WHEN the Field Ownership step renders THEN the system SHALL read the `fieldOwnershipMap` from the pipeline response and render each node's fields grouped by category, showing the fill-mode toggle (AI at runtime / static value) for each field; fields with no AI default SHALL be rendered as disabled/locked until the user explicitly enables them.

2.11 WHEN the ComprehensiveQuestions UI step renders its question list THEN the system SHALL exclude any question whose `type` is `credentialId` or whose `category` is `credentials`, so that only node-field configuration questions appear in this step.

### Unchanged Behavior (Regression Prevention) â€” UI Layer

3.9 WHEN the user completes the ComprehensiveQuestions step THEN the system SHALL CONTINUE TO transition to the "Refining Workflow Plan" loading state and invoke the backend refinement call with the user's answers.

3.10 WHEN the Credentials step renders THEN the system SHALL CONTINUE TO show OAuth and vault credential prompts (including the `credentialId` questions excluded from the questions step), and credential values entered by the user SHALL CONTINUE TO be pushed directly to the workflow.

3.11 WHEN a field in the Field Ownership step has a `buildtime_ai_once` or `runtime_ai` fill mode THEN the system SHALL CONTINUE TO pre-fill that field's display value with the AI-generated default from the pipeline response, and the user SHALL CONTINUE TO be able to override it with a static value.
