# Bugfix Requirements Document

## Introduction

AI pre-built values generated during the `buildtime_ai_once` phase (e.g. form fields, if/else conditions, Slack message text/channel) are not persisting through the credential configuration workflow. Each call to the `attach-inputs` endpoint re-runs normalization and field injection, but the `_fillMode` guard that is supposed to protect these values (`shouldPreserveExistingBuildtimeValue`) never fires because the `_fillMode` metadata is not stamped onto the node config during AI generation. As a result, `appliedFields: []` appears in every fill-mode snapshot and the AI-built values are silently overwritten or reset to defaults.

## Bug Analysis

### Current Behavior (Defect)

1.1 WHEN the `property-population-stage` writes AI-generated values (e.g. `fields`, `conditions`, `text`, `channel`) into `node.data.config`, THEN the system does NOT write the corresponding `_fillMode[fieldName] = 'buildtime_ai_once'` entries alongside those values.

1.2 WHEN `attach-inputs` is called for a node whose AI-built fields have no explicit `_fillMode` entry, THEN the system resolves the effective fill mode as `manual_static` (the fallback) instead of `buildtime_ai_once`, causing `shouldPreserveExistingBuildtimeValue` to return `{ preserve: false }` for every field.

1.3 WHEN `shouldPreserveExistingBuildtimeValue` returns `{ preserve: false }` for an AI-built field, THEN the system overwrites the existing AI-built value with the incoming (empty or default) value from the `attach-inputs` request, producing `appliedFields: []` in the fill-mode snapshot.

1.4 WHEN `attach-inputs` is called multiple times (once per node during credential configuration), THEN the system re-runs normalization on each call, and any AI-built value that survived the first call may be overwritten on a subsequent call because the `_fillMode` guard is still absent.

1.5 WHEN the `structure-materializer` stamps `_fillMode` entries for structural fields, THEN the system only stamps entries for fields identified as `isStructuralOwnership`, leaving non-structural `buildtime_ai_once` fields (e.g. Slack `text`, `channel`, `username`) without any `_fillMode` entry.

### Expected Behavior (Correct)

2.1 WHEN the `property-population-stage` writes an AI-generated value for a field whose `fillMode.default === 'buildtime_ai_once'`, THEN the system SHALL also write `config._fillMode[fieldName] = 'buildtime_ai_once'` into the same node config object.

2.2 WHEN `attach-inputs` processes a node field that has `_fillMode[fieldName] === 'buildtime_ai_once'` and a non-empty existing value, THEN the system SHALL invoke `shouldPreserveExistingBuildtimeValue` and receive `{ preserve: true }` when the incoming value is empty or weaker than the stored value.

2.3 WHEN `shouldPreserveExistingBuildtimeValue` returns `{ preserve: true }` for a field, THEN the system SHALL skip overwriting that field and record it in `modeDiagnostics.buildtimeMergePreserved`, leaving the AI-built value intact.

2.4 WHEN `attach-inputs` is called multiple times for the same workflow, THEN the system SHALL preserve all `buildtime_ai_once` fields that were set during AI generation across every call, as long as the user has not explicitly switched the field's fill mode to `manual_static`.

2.5 WHEN the `structure-materializer` stamps `_fillMode` entries, THEN the system SHALL stamp `_fillMode[fieldName] = 'buildtime_ai_once'` for every field in the node's `inputSchema` whose `fillMode.default === 'buildtime_ai_once'`, not only structural-ownership fields.

### AI-Built Values Visibility in Configuration Step (New Requirements)

4.0 WHEN the user reaches the Configuration/Credentials step for a workflow that was generated by the AI builder, THEN the system SHALL display the AI-built values pre-populated in every relevant input field, so the user can see exactly what the AI set before deciding whether to keep, edit, or replace them.

4.0.1 WHEN a node config field has `_fillMode[fieldName] === 'buildtime_ai_once'` and a non-empty stored value, THEN the Configuration UI SHALL render that stored value as the current field value — not an empty placeholder — so the user sees the AI's choice as the starting point.

4.0.2 WHEN the user views a pre-populated AI-built field in the Configuration step, THEN the system SHALL visually indicate that the value was set by the AI (e.g. the "AI (build)" pill is active) so the user understands the origin of the value.

4.0.3 WHEN the user decides to keep the AI-built value unchanged and submits the Configuration step, THEN the system SHALL preserve that value exactly as-is through the `attach-inputs` pipeline without modification.

4.0.4 WHEN the user edits an AI-built field value in the Configuration step (partial update), THEN the system SHALL accept the modified value, store it, and update `_fillMode[fieldName] = 'manual_static'` to reflect that the user has taken ownership of that field.

4.0.5 WHEN the user completely replaces an AI-built field value with a new value in the Configuration step, THEN the system SHALL store the new value and update `_fillMode[fieldName] = 'manual_static'`, discarding the original AI-built value.

4.0.6 WHEN the user clears an AI-built field value entirely in the Configuration step, THEN the system SHALL treat the empty submission as an explicit user action, store the empty value, and update `_fillMode[fieldName] = 'manual_static'` — it SHALL NOT silently restore the AI-built value.

4.0.7 WHEN the Configuration step is submitted, THEN the system SHALL send the current field value AND the effective fill mode (`mode_<nodeId>_<fieldName>`) together in the `attach-inputs` request, so the backend can correctly persist both the value and the ownership metadata.

### User Override & Fill Mode Switching (New Requirements)

4.1 WHEN a user clicks the "You" (manual) button on a field that currently has `_fillMode[fieldName] === 'buildtime_ai_once'`, THEN the system SHALL accept the user's value, write it into `node.data.config[fieldName]`, and update `_fillMode[fieldName] = 'manual_static'` — permanently overriding the AI-built value for that field.

4.2 WHEN `attach-inputs` receives a `mode_<nodeId>_<fieldName> = 'manual_static'` key alongside a user-supplied value for a field that was previously `buildtime_ai_once`, THEN the system SHALL apply the user-supplied value and persist `_fillMode[fieldName] = 'manual_static'` so that subsequent `attach-inputs` calls do not revert the field back to the AI-built value.

4.3 WHEN a user clicks the "AI (runtime)" button on ANY input field (regardless of its current fill mode — `manual_static`, `buildtime_ai_once`, or `runtime_ai`), THEN the system SHALL update `_fillMode[fieldName] = 'runtime_ai'` for that field and clear the stored static value, signalling that the AI will resolve this field dynamically at execution time.

4.4 WHEN `attach-inputs` receives a `mode_<nodeId>_<fieldName> = 'runtime_ai'` key for any field, THEN the system SHALL write `_fillMode[fieldName] = 'runtime_ai'` into the node config and SHALL NOT apply any static value for that field, regardless of what value is present in the request body.

4.5 WHEN a field's `_fillMode` is `runtime_ai` and `attach-inputs` is called again without an explicit mode override for that field, THEN the system SHALL preserve `_fillMode[fieldName] = 'runtime_ai'` and leave the field value empty/unset, so the AI can fill it at runtime.

4.6 WHEN a user switches a field from `runtime_ai` back to `manual_static` (by clicking "You" and providing a value), THEN the system SHALL accept the user-supplied value, write it to config, and update `_fillMode[fieldName] = 'manual_static'`.

4.7 WHEN a user switches a field from `manual_static` to `runtime_ai` (by clicking "AI (runtime)"), THEN the system SHALL update `_fillMode[fieldName] = 'runtime_ai'` and the previously stored manual value SHALL be cleared from the node config.

4.8 WHEN the fill mode toggle UI renders a field, THEN the system SHALL display the current effective fill mode (You / AI build / AI runtime) based on the `_fillMode[fieldName]` value stored in the node config, so the UI accurately reflects the persisted state after page reload or re-navigation.

### Form Fields Array Truncation (New Bug — Separate Root Cause)

5.1 WHEN the wizard pre-populates the `fields` question for a form node, THEN the system converts the AI-generated `fields` array to a JSON string via `snapshotConfigFieldToString` and stores it in `inputValues[questionKey]`.

5.2 WHEN the user submits the Configuration step without changing the `fields` value, THEN the wizard sends the JSON string (e.g. `'[{"id":"studentName",...},{"id":"marks",...}]'`) as the value for `config_node_<id>_fields` in the `attach-inputs` request.

5.3 WHEN `attach-inputs` receives a non-empty JSON string for the `fields` field, THEN `shouldPreserveExistingBuildtimeValue` returns `{ preserve: false }` (because the incoming value is non-empty), and the backend stores the string as-is — overwriting the original array with a string.

5.4 WHEN `PropertiesPanel` reads `config.fields` after the wizard completes, THEN `Array.isArray(config.fields)` returns `false` (because it is now a string), so `fields` is coerced to `[]` and the form builder shows no fields — or shows only the last field if partial parsing occurs.

5.5 WHEN `attach-inputs` receives a string value for a field whose `inputSchema` type is `array` or `object`, THEN the system SHALL attempt to `JSON.parse` the string and store the parsed value (array/object) instead of the raw string, so the backend always stores the correct type.

5.6 WHEN `JSON.parse` fails for a string value on an array/object field, THEN the system SHALL log a warning and skip applying the value (do not store a malformed string), leaving the existing stored value intact.

5.7 WHEN the wizard sends a JSON string for a `buildtime_ai_once` array/object field that is identical in content to the stored array/object (after parsing), THEN `shouldPreserveExistingBuildtimeValue` SHALL treat the parsed incoming value as equal to the existing value and return `{ preserve: true }` — preventing unnecessary overwrites.

### Unchanged Behavior (Regression Prevention)

3.1 WHEN a user explicitly sets a field's fill mode to `manual_static` via a `mode_<nodeId>_<fieldName>` key in the `attach-inputs` request, THEN the system SHALL CONTINUE TO accept and apply the user-supplied value for that field, overriding any prior AI-built value.

3.2 WHEN `attach-inputs` receives a non-empty incoming value for a `buildtime_ai_once` field and the existing stored value is empty, THEN the system SHALL CONTINUE TO apply the incoming value (no false preservation of empty defaults).

3.3 WHEN `attach-inputs` processes credential-owned fields (e.g. OAuth tokens, webhook URLs), THEN the system SHALL CONTINUE TO route those fields through the existing credential guard (`shouldApplyCredentialOwnedFieldViaAttachInputs`) unchanged.

3.4 WHEN `attach-inputs` processes switch `cases` or `rules` fields, THEN the system SHALL CONTINUE TO allow the user to reduce the number of cases freely (the `STRUCTURAL_BRANCH_FIELDS` exemption in `shouldPreserveExistingBuildtimeValue` must remain in effect).

3.5 WHEN the `property-population-stage` encounters a node type not found in the registry, THEN the system SHALL CONTINUE TO skip that node without error.

3.6 WHEN the `structure-materializer` runs in post-freeze readonly mode, THEN the system SHALL CONTINUE TO return the workflow unchanged without stamping any `_fillMode` entries.
